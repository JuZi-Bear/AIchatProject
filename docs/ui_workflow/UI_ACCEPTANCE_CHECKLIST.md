# Web UI 优化验收清单

用于后续每一轮 UI/UX 优化后检查页面是否真正变得更适合比赛展示。

## 页面顺序

- [ ] 页面顺序是否清晰。
- [ ] 首屏是否先展示状态摘要和工作流。
- [ ] Agent 输出是否放在关键结果之后。
- [ ] 报告和历史记录是否放在页面底部。
- [ ] Header 是否同时展示项目名称、当前模型、运行状态和 run_id。
- [ ] 最终结果总览是否位于 Agent 输出详情之前。
- [ ] 结果索引是否提供固定查看入口。

## 首屏核心结果

- [ ] 首屏是否能看到核心结果。
- [ ] 首屏是否能看到最终结果。
- [ ] 是否能一眼看到成功/失败。
- [ ] 是否能一眼看到修复次数。
- [ ] 是否能一眼看到当前 Agent。
- [ ] 是否能一眼看到质量评分。
- [ ] 是否能一眼看到当前模型。
- [ ] 是否能一眼看到 pytest 是否通过。
- [ ] 是否能一眼看到覆盖率和安全状态。

## 工作流可视化

- [ ] Waiting / Running / Done / Failed / Repairing 是否有明显区分。
- [ ] 当前执行 Agent 是否高亮。
- [ ] 自动修复链路是否有黄色提示。
- [ ] 最终成功节点是否有绿色强调。

## 空白与信息密度

- [ ] 页面是否整体紧凑。
- [ ] 是否减少多余空白。
- [ ] 是否避免大面积空 box。
- [ ] 是否减少重复容器。
- [ ] 是否避免卡片套卡片。
- [ ] 长文本是否默认折叠。
- [ ] 摘要是否控制在 1-3 行。
- [ ] 短字段是否使用 columns 或紧凑横向卡片展示。
- [ ] 工作流节点是否只展示简短摘要而不是长日志。
- [ ] success、report_path、stdout/stderr 是否避免在多个大区域重复展示。

## 结果查找

- [ ] 是否减少查找结果步骤。
- [ ] 三步内能找到最终代码。
- [ ] 三步内能找到错误摘要。
- [ ] 三步内能找到报告。
- [ ] 是否能通过结果索引快速进入最终代码。
- [ ] 是否能通过结果索引快速进入测试结果。
- [ ] 是否能通过结果索引快速进入错误摘要。
- [ ] 是否能通过结果索引快速进入插件结果。
- [ ] 是否能通过结果索引快速进入报告。
- [ ] 是否能通过结果索引快速进入自动修复过程。
- [ ] 是否能通过结果索引快速定位历史记录入口。
- [ ] stdout 是否有固定入口。
- [ ] stderr 是否有固定入口。
- [ ] pytest 结果是否独立可见。
- [ ] 插件结果是否有摘要和详细展开。
- [ ] 报告是否能快速打开和下载。

## 演示模式

- [ ] 演示模式是否隐藏完整 prompt。
- [ ] 演示模式是否隐藏完整 state。
- [ ] 演示模式是否隐藏过长 stderr。
- [ ] 演示模式是否保留错误摘要。
- [ ] 演示模式是否突出自动修复高光。

## 开发模式

- [ ] 是否保留开发模式完整日志。
- [ ] 是否能查看完整 stdout。
- [ ] 是否能查看完整 stderr。
- [ ] 是否能查看完整 state。
- [ ] 是否能查看完整代码和 pytest 测试。
- [ ] 是否能查看完整插件结果。
- [ ] 完整信息是否默认折叠而不是直接铺满页面。

## 回归检查

- [ ] 简单成功案例可运行。
- [ ] 翻车修复案例可触发自动修复。
- [ ] 综合案例可生成报告。
- [ ] 插件开关仍可写回配置。
- [ ] 历史记录仍可查看。
- [ ] Docker 启动方式不受影响。
