# Web UI 系统化优化计划

## 当前问题总结

当前 Web UI 已经具备完整演示能力，但在比赛现场展示时还存在以下体验问题：

- 展示顺序不够聚焦，用户需要在多个区域之间来回查找结果。
- 状态卡片、Agent 输出、测试结果、报告和历史记录之间层级不够清晰。
- 部分 box 留白较多，信息密度偏低，投屏时核心结果不够集中。
- Agent 工作流运行过程缺少持续视觉引导，观众不容易判断当前执行到哪个节点。
- 自动修复过程虽然有结果，但缺少更强的视觉“发生了什么”的瞬间提示。
- 开发模式和演示模式的信息边界还可以更明确。
- 插件、报告、历史记录等结果位置偏后，查找链路较长。

## 优化目标

- 首屏直接回答三个问题：是否成功、当前执行到哪一步、关键结果是什么。
- 让比赛评委不用滚动太多就能看到需求、工作流、成功/失败、修复次数和质量评分。
- 将“多 Agent 协作”和“自动修复闭环”变成页面视觉主线。
- 减少重复卡片和空白 box，提升信息密度。
- 保留开发模式完整日志，避免为了演示效果牺牲调试能力。
- 优化结果查找路径，让 stdout、stderr、pytest、插件结果和报告都有固定入口。

## 优化原则

- 不引入复杂前端框架，只使用 Streamlit 原生组件和少量 CSS。
- 不改变 LangGraph 核心流程，不改变 Agent 业务逻辑。
- 演示模式优先展示摘要，开发模式保留完整内容。
- 页面结构按“控制 → 运行状态 → 工作流 → 关键结果 → 详细日志 → 报告历史”组织。
- 一个区域只承担一个主要目的，避免同一信息在多个位置重复出现。
- 所有状态文案使用中文，必要时保留英文状态作为辅助。

## 页面信息优先级

第一优先级，首屏必须看到：

- 当前运行状态
- success 成功/失败
- 当前执行 Agent
- retry_count 修复次数
- quality_score 质量评分
- 用户需求摘要
- Agent 工作流节点

第二优先级，首屏或首屏下方可见：

- 自动修复高光时刻
- pytest 测试结果
- coverage 覆盖率
- stdout 摘要
- stderr 错误摘要
- 插件结果摘要

第三优先级，折叠或 Tabs 中查看：

- 完整代码
- 完整 pytest 代码
- 完整 stdout / stderr
- 完整 state
- 完整插件输出
- 历史运行记录
- Markdown 报告全文

## 分阶段改造计划

### 第一阶段：信息架构整理

- 固定主页面展示顺序。
- 合并重复状态卡片。
- 将最终结果、质量评分和修复次数提升到首屏。
- 将 Agent 输出 Tabs 放在关键摘要之后。

### 第二阶段：布局密度优化

- 减少大面积空白 box。
- 将横向卡片改成更紧凑的网格。
- 限制长文本默认高度，使用 expander 展开。
- 将报告和历史记录放入底部独立区域。

### 第三阶段：工作流视觉强化

- 当前 Agent 使用高亮边框和背景。
- Running 节点显示 spinner 或动态文案。
- Repairing 节点显示黄色提示。
- 自动修复发生时显示高光卡片。

### 第四阶段：结果查找路径优化

- 增加“最终结果总览”区域。
- stdout、stderr、pytest、报告使用固定入口。
- 演示模式只显示摘要，开发模式一键展开完整日志。

### 第五阶段：验收与回归

- 对简单案例、翻车修复案例、综合案例分别截图检查。
- 检查 Web UI、CLI、插件、报告、历史记录不受影响。
- 检查移动/窄屏时文字不溢出。
- 检查开发模式仍可查看完整 state 和日志。
