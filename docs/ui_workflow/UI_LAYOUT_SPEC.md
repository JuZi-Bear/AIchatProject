# Web UI 布局规格

## 推荐页面布局

页面采用“左侧控制栏 + 右侧主展示区”的稳定结构。

```text
┌──────────────────────┬──────────────────────────────────────────────┐
│ 左侧控制区            │ 顶部状态总览                                  │
│                      ├──────────────────────────────────────────────┤
│ 案例 / 模型 / 插件    │ Agent 工作流进度                              │
│ 运行模式 / 审批       ├──────────────────────────────────────────────┤
│ 开始 / 清空           │ 关键结果区：成功状态、修复、高光、质量评分       │
│                      ├──────────────────────────────────────────────┤
│                      │ 详细输出 Tabs：Agent / 测试 / 插件 / 日志       │
│                      ├──────────────────────────────────────────────┤
│                      │ 报告区 + 历史记录区                            │
└──────────────────────┴──────────────────────────────────────────────┘
```

## 展示顺序

第三步重排后的右侧主展示区固定为七层：

1. Header 区：项目名称、当前模型、当前运行状态、当前 run_id。
2. 控制区 + 状态摘要区：左侧 sidebar 保留控制项，主区域顶部横向展示 success、retry_count、test_success、coverage_percent、quality_score、security_status。
3. AI 工作流进度区：基于 `ui_view_model["workflow_steps"]` 按 order 展示 Requirement、Product、Coder、Tester、Approval、Runner、Sentry、Plugins、Quality、Report，每个节点显示名称、状态和简短摘要。
4. 最终结果总览区：集中展示成功 / 失败、修复次数、测试是否通过、覆盖率、质量评分、报告路径和错误摘要。
5. Agent 输出详情 Tabs：Product、Coder、Tester、Sentry、Plugins、Report、Raw State。Raw State 只在开发模式使用。
6. 结果索引区：提供查看最终代码、测试结果、错误摘要、插件结果和报告的固定入口。
7. 历史记录区：放在页面底部或侧边栏下方，不干扰主流程演示。

演示模式只展示 Header、状态摘要、工作流进度、自动修复高光、最终结果总览和报告入口。开发模式展示完整 Tabs、结果索引和历史记录。

## 左侧控制区设计

左侧控制区只放“运行前和运行中需要操作”的内容：

- 演示案例选择。
- 自定义需求输入。
- 模型选择。
- 多模型对比开关。
- 最大修复次数。
- 演示模式 / 开发模式。
- 在线 / 离线状态。
- 插件开关。
- 人工审批 checkbox。
- 开始运行按钮。
- 清空结果按钮。

控制区不展示长结果，不放完整日志。

## 主展示区设计

主展示区承担“看过程”和“看结果”：

- 顶部放最关键状态：成功/失败、当前 Agent、修复次数、质量评分、模型。
- 中部放 Agent 工作流节点，作为整页视觉主线。
- 关键结果区紧跟工作流，让观众立即看到结果。
- 详细输出放 Tabs，避免页面过长。

## 运行结果区设计

运行结果区应固定展示：

- 最终状态：成功 / 失败 / 等待人工确认。
- 修复次数。
- pytest 是否通过。
- coverage 覆盖率。
- quality_score。
- stdout 摘要。
- stderr 摘要。

完整 stdout / stderr 放入 expander。

## 报告区设计

报告区放在详细输出之后，作为结果归档入口：

- 最新报告文件名。
- 报告状态：成功 / 失败 / 未生成。
- 查看报告内容 expander。
- 下载 Markdown 报告按钮。
- 如果没有报告，显示“暂无报告”。

报告区不要重复展示完整 Agent 输出，只预览 Markdown 正文。

## 历史记录区设计

历史记录区放在页面底部：

- 历史 run_id 下拉选择。
- 成功状态、修复次数、模型、报告路径摘要。
- 查看历史代码 expander。
- 查看历史错误日志 expander。
- 加载为当前结果按钮。

历史记录默认折叠，避免干扰现场主流程。

## 如何减少空白和重复 box

- 将同类指标合并为一行 4-5 个紧凑卡片。
- 状态摘要使用 6 个紧凑横向卡片，不再把模型、状态、插件和结果混在同一组大卡片中。
- 不给每段文本都套大卡片，长文本优先使用 expander。
- Agent 输出只在 Tabs 中展示一次，摘要卡片只展示 1-2 行。
- 空状态使用一行提示，不使用大面积空白面板。
- 每个 box 保持明确标题，避免“卡片里套卡片”。
- 控制卡片高度，长文本使用摘要函数截断。
- 运行前隐藏或折叠详细结果区域，只展示引导提示。
- stdout、stderr、pytest 日志和报告 Markdown 默认折叠。
- 历史记录放到底部，避免比赛演示时观众先看到旧运行数据。
