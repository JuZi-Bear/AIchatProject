# 推荐扩展路线

本文给出当前阶段最适合的技术框架扩展路线。核心原则是先稳平台骨架，再做复杂平台能力。

## 当前最推荐先做

### 1. Java 任务生命周期管理

原因：

- Java 已经保存运行记录和报告索引。
- 下一步最自然的是把运行记录升级为任务生命周期。
- 可增强平台能力，又不影响 v1.0 Streamlit。

建议边界：

- 先定义状态：created、running、success、failed、cancelled。
- 不引入队列系统。
- 不改变 Python `/runs` 返回结构。

### 2. Java 配置中心增强

原因：

- 当前已有 settings、models、plugins 的 MySQL 持久化雏形。
- 可以先统一平台配置读取和保存逻辑。

建议边界：

- yaml 继续作为 Python Direct 兜底配置。
- Java 配置中心只在 Java Gateway 模式优先使用。

### 3. Vue 接入平台配置

原因：

- Vue 已支持 API 模式切换和 settings store。
- 平台配置接入能直接提升 v2 使用体验。

建议边界：

- 页面展示配置来源。
- 配置接口失败时回退 localStorage。
- 不删除 Python Direct。

### 4. Python Agent 注册中心

原因：

- Agent 扩展会越来越多，需要统一描述和注册。
- 可以先做只读注册中心，不改变默认工作流。

建议边界：

- 先记录 Agent name、label、description、enabled、stage。
- 不大改 LangGraph 主流程。

### 5. WebSocket / SSE 实时日志预留

原因：

- 平台工作台需要更好的运行过程展示。
- 可以先规划接口和数据格式，不立刻实现完整异步队列。

建议边界：

- 先写 API 契约和前端占位。
- 不引入队列系统。
- 不破坏当前同步 `/runs`。

## 当前不建议马上做

### 1. 用户系统

原因：会引入认证、会话、密码、安全和数据隔离，不适合在平台骨架未稳定前加入。

### 2. 权限系统

原因：权限依赖用户、项目空间和资源模型，过早做会造成大量抽象。

### 3. 团队协作

原因：依赖用户系统、项目空间、权限和审计日志，应放在平台基础稳定之后。

### 4. 工作流拖拽编辑器

原因：需要稳定 Workflow 模板、节点协议和执行 DSL，过早做会迫使 LangGraph 主流程大改。

### 5. 本地模型完整接入

原因：本地模型涉及硬件、推理服务、模型格式和性能，不适合作为当前首要框架扩展。

### 6. C++ Runner 完全替代 Python Runner

原因：C++ Runner 当前仍是最小版本，不具备完整隔离。应继续保持可选增强和 Python fallback。

## 路线选择理由

- 当前阶段还要保持双轨稳定。
- 不应过早引入权限和复杂协作。
- 应先把平台骨架稳定下来。
- 先增强 Java 平台层、Vue 平台配置和 Python Agent 扩展边界，再进入多用户和可视化编排。
