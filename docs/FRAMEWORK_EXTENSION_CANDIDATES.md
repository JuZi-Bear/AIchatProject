# 技术扩展候选清单

本文列出当前项目可考虑的技术框架扩展候选项，并按优先级、收益、风险和是否建议现在做进行评估。

| 扩展项 | 所属层 | 优先级 | 收益 | 风险 | 是否建议现在做 |
|---|---|---|---|---|---|
| Java 任务生命周期管理 | Java 平台服务层 | P1 | 让任务从创建、运行、完成、失败、重试到归档可管理 | 状态设计过早复杂化 | 是，先做最小生命周期 |
| Java 配置中心 | Java 平台服务层 | P1 | 统一管理前端设置、模型、插件和运行参数 | 与 yaml 配置产生双源冲突 | 是，保持 yaml 兜底 |
| Java 用户系统 | 平台化能力 | P3 | 支持多用户和身份区分 | 引入认证、会话和权限复杂度 | 否 |
| Java 项目空间管理 | 平台化能力 | P3 | 支持多项目隔离和团队协作基础 | 需要用户系统和数据隔离 | 否 |
| MySQL 数据表规范化 | 数据层 | P2 | 提升查询、统计和扩展能力 | 可能需要迁移和兼容旧数据 | 暂缓，先稳定现有表 |
| Vue 工作流可视化编辑器 | 前端平台 | P3 | 展示强，适合平台化愿景 | 会推动后端 Workflow 模板大改 | 否 |
| Vue 报告中心增强 | 前端平台 | P2 | 提升报告检索、预览和演示体验 | 依赖报告索引质量 | 是，可小步增强 |
| Vue 模型评测页面 | 前端平台 | P2 | 展示多模型对比能力 | 需要稳定评测数据结构 | 可规划，暂不急 |
| Python Agent 注册中心 | Python Agent Engine | P1 | 让 Agent 扩展更规范，减少硬编码 | 可能触及 Agent 调用链 | 是，先做只读注册描述 |
| Python Prompt 模板管理 | Python Agent Engine | P2 | 方便维护和调参 | 改 Prompt 会影响输出稳定性 | 可规划，不建议立即迁移核心 Prompt |
| 多模型 Provider 抽象层 | Python Agent Engine | P2 | 更好接入 DeepSeek/Qwen/GLM 等 Provider | 容易影响现有模型调用 | 可规划，先补接口边界 |
| 本地模型适配 | Python Agent Engine | P3 | 支持离线和私有化部署 | 环境差异大，推理资源复杂 | 否 |
| C++ Runner 正式接入 | 执行安全 | P2 | 提升执行安全和工程亮点 | 当前隔离能力不足 | 暂缓完全接入，继续增强 fallback |
| Docker 沙箱执行 | 执行安全 | P3 | 更强隔离边界 | 资源、性能和平台兼容复杂 | 否 |
| WebSocket / SSE 实时运行日志 | API / 前端 | P1 | 提升运行过程可视化，适合平台工作台 | 需要异步状态和日志缓冲 | 是，先做协议预留 |
| 队列系统 | 平台服务 | P3 | 支持异步任务和并发执行 | 引入 Redis/MQ 等新依赖 | 否 |
| 审计日志 | 平台服务 | P2 | 记录配置、运行和安全事件 | 需要定义事件模型 | 可规划，用户系统前先做轻量事件 |
| 权限系统 | 平台化能力 | P3 | 支持企业化访问控制 | 需要用户系统和资源模型 | 否 |
| 多用户团队协作 | 平台化能力 | P3 | 支持团队使用和项目空间 | 复杂度高，依赖用户/权限 | 否 |
| 录屏 / 演示模式增强 | 前端 / 演示 | P2 | 提升比赛展示效果 | 容易变成 UI 单点优化 | 可小步做，不应优先于平台骨架 |
